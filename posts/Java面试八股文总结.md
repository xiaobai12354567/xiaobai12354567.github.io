# Java面试八股文总结

## 一、JavaSE

### 1. 接口与抽象类的区别

- **定义**：
  - 抽象类：用 `abstract` 修饰的类，可以包含抽象方法和具体方法，可以有成员变量、构造方法。
  - 接口：用 `interface` 定义，JDK8之前只能有抽象方法，JDK8开始可以有默认方法（`default`）和静态方法，JDK9允许私有方法。

- **区别**：
  - **单继承 vs 多实现**：类只能继承一个抽象类，但可以实现多个接口。
  - **成员变量**：抽象类可以有实例变量、静态变量；接口中变量默认是 `public static final` 常量。
  - **构造方法**：抽象类有构造器，用于子类初始化；接口没有构造器。
  - **访问修饰符**：抽象类方法可以有各种修饰符；接口方法默认 `public`（抽象方法），JDK8默认方法也是 `public`。
  - **设计思想**：抽象类体现“is-a”关系，模板方法设计；接口体现“like-a”关系，能力约定。

## 二、集合

### 2. HashMap数据结构是什么？为什么HashMap在JDK1.8引入红黑树？HashMap扩容机制？

- **数据结构**：数组 + 链表（JDK1.7），JDK1.8及以后改为数组 + 链表 + 红黑树。当链表长度超过阈值（默认8）且数组长度大于等于64时，链表转为红黑树。
- **引入红黑树原因**：为了解决哈希冲突严重时链表过长导致查询效率低下的问题（链表查询O(n)），红黑树查询O(log n)提升性能。
- **扩容机制**：
  - 初始容量16，负载因子0.75，当元素个数超过 `容量 * 负载因子` 时触发扩容。
  - 扩容为原容量的2倍，并重新计算每个元素在新数组中的位置（rehash）。
  - JDK1.8优化：元素在新数组中的索引要么是原位置，要么是原位置+原容量，利用与运算代替取模。

### 3. 为什么数组长度为2的幂次方？为什么JDK7到JDK8要把头插改尾插？为什么解决hash冲突的方式为链表+红黑树？

- **数组长度2的幂次方原因**：
  - 计算索引时使用 `hash & (length-1)` 等价于取模运算，且速度更快。
  - 保证哈希值均匀分布，减少碰撞。
- **头插改尾插原因**：
  - JDK1.7头插法在多线程扩容时可能形成循环链表（死循环）。
  - JDK1.8改为尾插法，避免死循环，且保持插入顺序。
- **链表+红黑树原因**：
  - 链表适用于少量冲突，插入删除快；当冲突严重时，红黑树提高查询效率。
  - 权衡空间和时间，红黑树节点占用空间大，所以仅在链表长度超过阈值时转换。

### 4. ArrayList与LinkedList区别？

- **底层结构**：ArrayList基于动态数组，LinkedList基于双向链表。
- **随机访问**：ArrayList支持高效随机访问（O(1)），LinkedList需要遍历（O(n)）。
- **插入删除**：ArrayList在中间插入删除需要移动元素（O(n)），LinkedList只需修改指针（O(1)），但需先定位位置。
- **内存占用**：ArrayList连续内存，预分配容量可能浪费；LinkedList每个节点有额外指针开销。
- **适用场景**：ArrayList适合读多写少、随机访问；LinkedList适合频繁插入删除、队列或栈场景。

### 5. ConcurrentHashMap如何实现？从JDK7到JDK8做了什么升级？

- **JDK7实现**：
  - 采用分段锁（Segment继承ReentrantLock），将数据分成多个Segment，每个Segment独立加锁，并发度等于Segment数量。
  - 数据结构：Segment数组 + HashEntry数组 + 链表。
- **JDK8升级**：
  - 摒弃分段锁，采用CAS + synchronized保证线程安全。
  - 数据结构：Node数组 + 链表 + 红黑树。
  - 锁粒度更细：只锁链表或红黑树的头节点，并发度更高。
  - 引入红黑树优化查询性能。
  - 使用CAS进行无锁操作（如插入、扩容协助），减少锁竞争。

## 三、JVM

### 6. JVM如何调优？有哪些垃圾回收算法？

- **JVM调优**：
  - 设置合理的堆内存大小（-Xms、-Xmx）。
  - 选择合适的垃圾收集器（如G1、CMS、Parallel）。
  - 调整新生代与老年代比例（-XX:NewRatio）。
  - 设置日志参数（-XX:+PrintGCDetails）分析GC情况。
  - 避免频繁Full GC，优化代码减少对象创建。
- **垃圾回收算法**：
  - **标记-清除**：先标记存活对象，再清除未标记对象，产生碎片。
  - **复制**：将内存分为两块，每次只用一块，存活对象复制到另一块，适合新生代（Minor GC）。
  - **标记-整理**：标记存活对象，然后向一端移动，清除边界外的内存，适合老年代。
  - **分代收集**：结合不同算法，新生代用复制，老年代用标记-整理或标记-清除。

### 7. JVM内存空间？

- **线程私有**：
  - 程序计数器：当前线程执行的字节码行号。
  - 虚拟机栈：每个方法调用对应一个栈帧，存储局部变量表、操作数栈、动态链接、方法出口。
  - 本地方法栈：为Native方法服务。
- **线程共享**：
  - 堆：存放对象实例，GC管理的主要区域，可分为新生代（Eden、Survivor0、Survivor1）和老年代。
  - 方法区：存储类信息、常量、静态变量、即时编译代码等。JDK8后元空间（MetaSpace）取代永久代，使用本地内存。

### 8. 逃逸分析？

- **概念**：分析对象动态作用域，判断对象是否逃逸出方法或线程。
- **作用**：
  - 栈上分配：如果对象不逃逸，可在栈上分配，随方法结束自动销毁，减少GC压力。
  - 同步消除：如果对象不逃逸，可去掉同步锁。
  - 标量替换：将对象拆散为成员变量分配在栈上。
- **开启参数**：`-XX:+DoEscapeAnalysis`（JDK默认开启）。

### 9. 如何避免内存溢出异常？

- **堆溢出**：调整堆大小（-Xmx），检查是否有大对象泄漏（如内存泄漏分析工具MAT）。
- **栈溢出**：减少递归深度，调整栈容量（-Xss）。
- **方法区溢出**：增加元空间大小（-XX:MaxMetaspaceSize），注意动态生成类过多。
- **直接内存溢出**：合理使用NIO，调整-XX:MaxDirectMemorySize。
- **通用方法**：优化代码，及时释放引用，使用软/弱引用，避免静态集合无限增长。

## 四、JUC

### 10. 什么是乐观锁？CAS实现原理？什么是悲观锁？synchronized和ReentrantLock的区别？

- **乐观锁**：假设没有冲突，更新时再检查，适用于读多写少。典型实现：CAS。
- **CAS原理**：比较并交换，包含三个操作数：内存位置V、预期值A、新值B。仅当V等于A时，才将V更新为B，否则重试或失败。硬件层面保证原子性。
- **悲观锁**：假设有冲突，先加锁再操作。如synchronized。
- **synchronized vs ReentrantLock**：
  - synchronized是关键字，自动释放锁；ReentrantLock是API，需手动解锁。
  - ReentrantLock可中断锁、可超时、可尝试获取锁（tryLock）、支持公平锁。
  - ReentrantLock可绑定多个条件（Condition）。
  - JDK6后synchronized性能优化（锁升级），两者性能相近。

### 11. synchronized锁升级流程？是否为可重入锁，可重入锁为了保证什么？

- **锁升级流程**：无锁 → 偏向锁 → 轻量级锁 → 重量级锁。
  - 偏向锁：只有一个线程访问时，记录线程ID，避免CAS。
  - 轻量级锁：多个线程交替竞争时，CAS自旋获取锁。
  - 重量级锁：竞争激烈时，阻塞线程，依赖操作系统互斥量。
- **可重入锁**：synchronized和ReentrantLock都是可重入的，即同一线程可重复获取同一把锁。
- **可重入作用**：避免死锁，允许递归调用，保证线程在持有锁的情况下再次获取锁不会阻塞。

### 12. 原子类如何实现？

- 原子类（如AtomicInteger）基于CAS（Unsafe类的compareAndSwapInt）实现。
- 内部使用volatile修饰value保证可见性，通过CAS循环更新直到成功。
- 提供原子操作如incrementAndGet、compareAndSet等。

### 13. volatile关键字的作用？

- **可见性**：保证不同线程对volatile变量的修改能立即被其他线程看到，禁止线程本地缓存。
- **有序性**：禁止指令重排序（通过内存屏障），防止JVM对代码优化导致的问题。
- **不保证原子性**：如i++操作仍需同步。

### 14. JMM内存模型？指令重排序？happens-before原则？

- **JMM**：Java内存模型，定义线程与主内存之间的抽象关系。每个线程有工作内存（缓存），操作变量需先拷贝到工作内存。
- **指令重排序**：编译器和处理器为优化性能而对指令顺序进行调整，但需遵守happens-before规则。
- **happens-before原则**：
  - 程序次序规则：单线程内，写在前面的操作先行发生于后面的。
  - volatile规则：对volatile变量的写先行发生于读。
  - 锁规则：解锁先行发生于加锁。
  - 传递性等。

### 15. AQS队列如何实现？如何实现公平锁？

- **AQS**：抽象队列同步器，核心是volatile int state（同步状态）和CLH双向队列（FIFO）。
- 线程获取锁失败时，包装成Node节点加入队列，通过自旋和LockSupport阻塞。
- **公平锁实现**：ReentrantLock公平锁在tryAcquire时，先检查队列中是否有等待线程，若有则入队，不插队。

### 16. 线程池的核心参数？提交任务流程？核心线程池如何计算为什么？

- **核心参数**：
  - corePoolSize：核心线程数
  - maximumPoolSize：最大线程数
  - keepAliveTime：空闲线程存活时间
  - unit：时间单位
  - workQueue：阻塞队列
  - threadFactory：线程工厂
  - handler：拒绝策略
- **提交任务流程**：
  1. 如果当前线程数 < corePoolSize，新建线程执行任务。
  2. 如果 >= corePoolSize，任务加入队列。
  3. 如果队列满且线程数 < maximumPoolSize，创建非核心线程执行任务。
  4. 如果队列满且线程数已达最大，执行拒绝策略。
- **核心线程数计算**：
  - CPU密集型：设为 `CPU核心数+1`，避免过多线程切换。
  - IO密集型：设为 `2 * CPU核心数` 或更多，因为IO等待时线程可让出CPU。

## 五、Spring

### 17. Spring Boot用了哪些设计模式？

- **工厂模式**：BeanFactory创建Bean。
- **单例模式**：Bean默认作用域为单例。
- **代理模式**：AOP实现。
- **模板方法**：JdbcTemplate、RedisTemplate等。
- **观察者模式**：ApplicationEvent事件机制。
- **策略模式**：Resource接口及各种实现。
- **适配器模式**：HandlerAdapter适配不同Controller。
- **装饰器模式**：BeanWrapper对Bean实例的包装。
- **责任链模式**：拦截器链。

### 18. IOC？

- **IoC（控制反转）**：将对象的创建和依赖关系的管理交给容器（Spring），而不是对象自己控制。即“好莱坞原则”：别调用我们，我们调用你。
- **DI（依赖注入）**：容器在创建对象时动态地将依赖注入到对象中。
- 优点：降低耦合，提高可测试性和可维护性。

### 19. 讲讲循环依赖在Spring如何解决的？

- **三级缓存**：
  - 一级缓存（singletonObjects）：存放完全初始化好的单例Bean。
  - 二级缓存（earlySingletonObjects）：存放提前暴露的Bean（未完全初始化，但已实例化）。
  - 三级缓存（singletonFactories）：存放Bean的工厂对象（ObjectFactory），用于生成代理对象。
- **解决过程**：
  1. A创建时，实例化后将ObjectFactory放入三级缓存。
  2. A依赖B，去创建B，B实例化后也放入三级缓存。
  3. B依赖A，从三级缓存获取A的ObjectFactory，生成A的早期引用放入二级缓存，并注入到B。
  4. B完成初始化，放入一级缓存。
  5. A继续完成初始化，从二级缓存获取早期引用，注入完成。
- 注意：只能解决单例、非构造器注入的循环依赖。

### 20. AOP实现原理？有哪些代理模式？运行代理还是编译代理？

- **实现原理**：基于动态代理，在运行时为目标对象生成代理对象，将增强逻辑织入。
- **代理模式**：
  - JDK动态代理：目标类实现接口时使用，通过InvocationHandler和Proxy生成代理对象。
  - CGLIB代理：目标类没有实现接口时使用，通过继承生成子类，拦截方法调用。
- **运行代理**：Spring AOP默认是运行时增强（动态代理）。还有编译时增强（AspectJ编译期织入）和类加载时增强。

### 21. Spring的事务什么情况会失效？

- 数据库引擎不支持事务（如MyISAM）。
- 方法非public（Spring默认只代理public方法）。
- 自调用（同一类内方法调用，不经过代理）。
- 异常被捕获未抛出。
- 异常类型不匹配（默认只回滚RuntimeException和Error，checked异常不回滚，需配置rollbackFor）。
- 传播行为设置不当。
- 没有被Spring管理（未加@Service等注解）。

## 六、Redis

### 22. Redis基本数据类型？常见应用场景

- **String**：字符串，可用于缓存、计数器、分布式锁。
- **Hash**：哈希表，存储对象（如用户信息）。
- **List**：双向链表，消息队列、最新列表。
- **Set**：无序集合，去重、共同好友、随机抽奖。
- **ZSet**（Sorted Set）：有序集合，排行榜、延时队列。
- **Bitmap**、**HyperLogLog**、**Geo**等扩展类型。

### 23. 缓存穿透？雪崩？击穿？

- **缓存穿透**：查询一个不存在的数据，缓存和数据库都没有，请求直接打到数据库。解决：缓存空对象、布隆过滤器。
- **缓存雪崩**：大量缓存同时失效，导致请求涌向数据库。解决：随机过期时间、加锁排队、集群高可用。
- **缓存击穿**：热点key过期，高并发访问该key瞬间压垮数据库。解决：互斥锁、热点数据永不过期。

### 24. 缓存数据一致性？

- 常见策略：
  - **先更新数据库，再删除缓存**（Cache Aside Pattern）。
  - **先删除缓存，再更新数据库**（可能导致不一致，需延时双删）。
  - **使用消息队列**异步同步。
  - **订阅数据库binlog**（如Canal）更新缓存。
- 强一致性很难，通常追求最终一致性。

### 25. Redis为什么快？解释I/O多路复用模型？

- **快的原因**：
  - 基于内存，读写速度快。
  - 单线程避免上下文切换和锁竞争。
  - 非阻塞I/O多路复用。
  - 数据结构高效。
- **I/O多路复用**：单个线程同时监听多个socket，通过select/epoll等系统调用，当有事件就绪时处理。Redis将连接、读取、写入等事件交给多路复用器，然后派发给文件事件分派器处理。

### 26. Redis实现登录机制？

- 通常用Redis存储登录状态，如：
  - 用户登录成功后生成token（如UUID），以token为key，用户信息为value存入Redis，设置过期时间。
  - 后续请求携带token，从Redis获取用户信息验证。
  - 可实现单点登录（同一账号只允许一个登录，踢掉之前）。

### 27. Redis实现防抖和节流？

- **防抖**：限制同一操作在短时间内重复触发。如提交订单，用Redis的setnx设置key，设置过期时间，若存在则拒绝。
- **节流**：限制单位时间内请求次数。可用Redis计数器（incr），并设置过期时间，超过阈值则限流。

### 28. 主从同步？持久化机制？

- **主从同步**：
  - 全量同步：从服务器连接主服务器，发送SYNC命令，主服务器执行BGSAVE生成RDB快照，并发送给从。
  - 增量同步：主服务器将写命令记录在缓冲区，异步发送给从（命令传播）。
  - 新版Redis使用PSYNC支持部分同步，断线重连后可恢复增量数据。
- **持久化机制**：
  - **RDB**：定期生成数据快照，文件小，恢复快，但可能丢失数据。
  - **AOF**：记录每个写命令，追加到文件，可通过重写压缩，数据更安全，但文件大，恢复慢。
  - 可混合使用。

### 29. bigkey和hotkey如何优化？

- **bigkey**（大key）：
  - 危害：内存不均、阻塞网络、删除耗时。
  - 优化：拆分（如hash分片）、压缩、使用其他数据结构。
- **hotkey**（热key）：
  - 危害：单节点压力大。
  - 优化：本地缓存（如JVM缓存）、读写分离、热key分散到多个副本。

## 七、MySQL

### 30. MySQL事务隔离级别？ACID？

- **隔离级别**：
  - 读未提交：脏读、不可重复读、幻读。
  - 读已提交：避免脏读，但可能不可重复读、幻读。
  - 可重复读（MySQL默认）：避免脏读、不可重复读，但可能幻读（InnoDB通过MVCC+间隙锁解决）。
  - 串行化：所有问题解决，但并发低。
- **ACID**：
  - 原子性：事务不可分割，要么全成功要么全失败。
  - 一致性：事务前后数据完整性约束。
  - 隔离性：并发事务相互隔离。
  - 持久性：事务提交后永久保存。

### 31. MVCC下，可重复读如何实现的？还有幻读出现吗？

- **MVCC实现**：通过隐藏字段（DB_TRX_ID、DB_ROLL_PTR）和undo log，以及Read View（活跃事务列表）。
  - 可重复读：事务启动时创建Read View，整个事务期间使用该视图，保证每次读取看到相同数据。
  - 快照读（普通select）通过MVCC避免幻读；当前读（select for update）通过间隙锁防止幻读。
- **幻读**：在可重复读下，快照读不会幻读，但当前读可能产生幻行，InnoDB通过间隙锁（Gap Lock）和临键锁（Next-Key Lock）解决。

### 32. 间隙锁？临键锁？回表查询如何避免？

- **间隙锁**：锁定一个范围（区间），防止其他事务插入，避免幻读。
- **临键锁**：行锁+间隙锁，锁定一个范围并锁定记录本身。
- **回表查询**：当使用非聚簇索引查询，且需要返回索引列以外的列时，需回表到聚簇索引获取数据。
- **避免回表**：使用覆盖索引（索引包含所有需要查询的字段）。

### 33. 常见索引？索引失效？

- **常见索引**：主键索引（聚簇）、唯一索引、普通索引、全文索引、组合索引。
- **索引失效场景**：
  - 违反最左前缀法则（组合索引）。
  - 范围查询（>、<）右边列失效。
  - 对索引列使用函数或计算。
  - 类型转换（字符串不加引号）。
  - 使用不等于（!= 或 <>）可能导致失效。
  - like以通配符开头（'%abc'）。
  - OR条件两边有非索引列。

### 34. explain关键字？

- 用于查看SQL执行计划，关键字段：
  - **id**：查询序号。
  - **select_type**：查询类型（SIMPLE、PRIMARY、SUBQUERY等）。
  - **table**：表名。
  - **type**：访问类型（system > const > eq_ref > ref > range > index > ALL）。
  - **possible_keys**：可能使用的索引。
  - **key**：实际使用的索引。
  - **key_len**：索引长度。
  - **rows**：预估扫描行数。
  - **Extra**：额外信息（Using index、Using where等）。

### 35. 索引数据结构？为何使用B+树？

- **数据结构**：B+树（InnoDB引擎）。
- **原因**：
  - 非叶子节点只存储键值，不存数据，可存储更多索引，降低树高，减少磁盘I/O。
  - 叶子节点形成有序链表，范围查询高效。
  - 所有数据都在叶子节点，查询稳定（I/O次数相同）。
  - 相比红黑树/二叉树，磁盘I/O更少；相比B树，B+树更有利于范围查询和排序。

### 36. MySQL三大日志

- **binlog**（二进制日志）：记录所有修改操作，用于主从复制、数据恢复。Server层实现。
- **redo log**（重做日志）：保证事务持久性，InnoDB特有。WAL技术，先写日志后写磁盘。崩溃恢复用。
- **undo log**（回滚日志）：用于事务回滚和MVCC，记录数据修改前的版本。

## 八、设计模式

### 37. 单例模式？双重锁检查？

- **单例模式**：确保一个类只有一个实例，并提供全局访问点。
- **双重锁检查（DCL）**：
  ```java
  public class Singleton {
      private static volatile Singleton instance;
      private Singleton() {}
      public static Singleton getInstance() {
          if (instance == null) {
              synchronized (Singleton.class) {
                  if (instance == null) {
                      instance = new Singleton();
                  }
              }
          }
          return instance;
      }
  }
  ```
  - 为什么用volatile？防止指令重排序，确保对象初始化完成后再赋值。
  - 优点：懒加载，线程安全，性能高。

- 其他实现方式：饿汉式、静态内部类、枚举。
